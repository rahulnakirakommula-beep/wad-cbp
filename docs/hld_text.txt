1. Introduction
1.1 Purpose
This High-Level Design (HLD) document describes the architectural decisions, component structure, data schemas, API contracts, background job specifications, and key data flows for the Opportunity Intelligence Platform (COA). It is the companion document to COA-SRS-001. Together they constitute the full technical specification for V1 development.

1.2 Scope
This document covers: overall system architecture, component responsibilities, MongoDB schema design (all eight collections), API endpoint inventory, background job logic, and three annotated key data flows. V2 expandability mappings are included in Section 9.

1.3 Architectural Principles
Index-first query design — every query pattern has a supporting MongoDB index.
Derived never stored — confidenceLevel and dataSourceYear are always computed at read time from raw date fields.
Event-driven critical notifications — don't-miss and cancellation notifications are dispatched immediately via event trigger, not cron.
Optimistic UI — all student mutations are reflected immediately in the UI with server-side reconciliation.
Audit everything — every state-changing operation writes an AuditLog entry with a diff object.
Additive V2 — no V1 schema or API contract changes are needed to add the documented V2 features.

2.3 Authentication Flow
JWT (7-day expiry) is stored in localStorage and sent as a Bearer token in the Authorization header. The Express auth middleware extracts and verifies the JWT on every protected route. The middleware attaches req.user to the request object. All admin routes additionally check req.user.role === 'admin'. Source routes check req.user.role === 'source'.

Email verification uses a short-lived JWT (24 h) embedded in the verification link. On click, the backend verifies the token, sets isEmailVerified: true, and invalidates the token (single-use enforced by a token blacklist or by deleting the token record).

6. Background Jobs
All cron jobs are implemented using node-cron and run within the Express process. Event-driven dispatches are triggered inline by the relevant API handler.
6.1 Status Transition Cron  —  Every hour
Step
Logic
1
Find listings where status='upcoming' AND openDate ≤ now. Batch update status='open'. AuditLog action: 'auto_opened' for each.
2
Find listings where status='open' AND deadline ≤ now. For each: if lastDeadline is null, copy deadline → lastDeadline. Set status='closed'. AuditLog action: 'auto_closed'.
3
Find UserActivity records where status='saved' AND the associated listing's deadline ≤ now. Batch update status='missed', statusUpdatedAt = listing.deadline (not current time). AuditLog.

6.2 Notification Dispatch Cron  —  Daily at 08:00
Step
Logic
1
3-day reminder: find UserActivity where status='saved' AND listing.deadline = today+3 days. For each user: check notificationPrefs.deadlineReminders = true AND no existing deadline_3day notification for this (userId, listingId) pair. Create Notification. If emailEnabled: send email.
2
1-day reminder: same logic for listing.deadline = today+1 day. type: deadline_1day. In-app notification only (no email).
3
Frequency cap enforcement: before creating any dont_miss notification, query Notifications where type='dont_miss' AND userId=X AND createdAt > 7 days ago. Skip if found.

6.3 Staleness Flag Cron  —  Every Sunday at 02:00
Find listings where status in [upcoming, open] AND (lastVerifiedAt < 270 days ago OR lastVerifiedAt is null). Set isStale: true. AuditLog: 'auto_flagged_stale'.
Find listings where timeline.lastDeadline is older than 2 years. Set isStale: true if not already set.

6.4 Season Window Cron  —  1st of every month at 09:00
Find listings where timeline.expectedStart = currentMonth AND status = 'upcoming'.
For each listing: find users with this listing's domainTags in their interests AND notificationPrefs.seasonAlerts = true AND no season_open notification for this listing received this year.
Cap: 1 per domain category per user per month. If user already has season_open for another listing in same category this month, skip.
Create Notification docs (type: season_open, in-app only).

6.5 Don't-Miss Event Dispatch  —  Event-driven
Triggered immediately when admin sets priority = 'dont-miss' via PUT /api/admin/listings/:id. Not a cron job.
Find all users where interests array intersects with listing's domainTags.
Apply frequency cap: skip users who received a dont_miss notification in the past 7 days.
Create Notification docs (type: dont_miss). In-app only.

6.6 Cancellation Event Dispatch  —  Event-driven
Triggered immediately when admin sets status = 'cancelled'. Not a cron job.
Query UserActivity for all users with saved or applied status for the cancelled listing.
Create Notification docs (type: cancelled) for all affected users.
In-app notification + email if user.notificationPrefs.emailEnabled = true.

6.7 Recommendation Pre-compute Job  —  Every 15–30 minutes
Runs as a background worker separate from the cron jobs (or as a frequently-scheduled cron). Ensures the Recommended feed section can be served from Redis without on-demand computation.
For each active user (status: active, onboardingComplete: true): run the eligibility filter + scoring query (Section FR-FEED-04 and FR-FEED-05 in the SRS).
Store top 8 results as JSON in Redis: key = rec:{userId}, TTL = 30 minutes.
On feed load, GET /api/feed/sections reads rec:{userId} from Redis. On cache miss, compute on demand and populate cache.
7. Key Data Flows
7.1 Student Discovers and Saves a Listing
#
Step
1
Student opens /app/feed. React Query fires three parallel calls: GET /api/feed/sections (reads recommended from Redis cache), GET /api/activity/summary, GET /api/notifications/unread-count.
2
Feed renders. Student sees 'Google STEP 2026' in the Recommended for You section with a score derived from matching domainTags.
3
Student taps the Save button. The UI immediately updates the card to 'Saved' state (optimistic update). POST /api/activity fires in the background.
4
Backend: verify JWT. Check UserActivity unique constraint — no existing record for this (userId, listingId). Create UserActivity { userId, listingId, status: 'saved', statusUpdatedAt: now, createdAt: now }. Response 201.
5
If the API call fails (network error, duplicate, etc.): the optimistic update rolls back. Card reverts to unsaved state. A toast error notification is shown.
6
That evening, the notification cron (08:00) checks if listing.deadline = today+3. If yes and user.notificationPrefs.deadlineReminders = true: create Notification { type: deadline_3day, payload snapshot, expiresAt: deadline }. Student sees badge on next visit.

7.2 Admin Publishes a New Listing
#
Step
1
Admin opens /admin/listings/new. Empty curation panel renders.
2
Admin fills all fields: title, description, type, timeline, targetAudience (years=[2,3], branches=[CSE,IT]), domainTags, status=open, priority=high, isCurated=true.
3
Admin clicks Save. POST /api/admin/listings. Backend: validate JWT + role:admin. Run auto-tagger (keyword suggestions from description — admin-set tags take precedence). Insert listing document with version=1.
4
AuditLog written: { category: 'listing', action: 'created_manually', actorType: 'user', actorId: adminUserId, diff: null }.
5
Listing is now live (status: open). Year-2 and year-3 CSE/IT students with frontend or matching interests will see it in Recommended on their next feed load (after the next recommendation pre-compute cycle).

7.3 Listing Completes Annual Cycle
#
Step
1
Status transition cron runs hourly. Detects listing.deadline ≤ now. Sets status='closed'. Copies deadline → lastDeadline (if null). Writes AuditLog: 'auto_closed'.
2
Simultaneously: finds all UserActivity records for this listing with status='saved'. Batch updates status='missed', statusUpdatedAt = listing.deadline. Writes AuditLog per record.
3
Listing disappears from student feed (feed query filters to status in [open, upcoming]). Students' tracking dashboards now show it under the Missed section.
4
Admin reviews the closed listing. Next year's dates are available. Admin opens the curation panel and clicks Cycle Reset. A confirmation dialog explains the action.
5
POST /api/admin/listings/:id/cycle-reset: atomic update — lastDeadline preserved (previous year's deadline), openDate and deadline cleared, status='upcoming', isCurated=false, version++. AuditLog: 'cycle_reset' with full diff.
6
Listing re-enters upcoming state. Admin fills in new year's dates and sets isCurated=true. Listing becomes active for the next season.

7.4 Organisation Submits a Listing (Review Flow)
#
Step
1
Organisation logs in to /org with their pre-registered credentials (role: source). Opens the Organisation Listing Manager.
2
Organisation fills in the new listing form and submits. POST /api/org/listings. Backend: create Listing with status='unknown', isCurated=false, sourceId pointing to the Organisation's Source record.
3
AuditLog: { category: 'listing', action: 'submitted_by_org', actorType: 'source' }. Listing enters the admin queue.
4
Admin sees +1 badge on the Submission Queue nav item. Opens /admin/queue. The new submission appears with status: unknown.
5
Admin clicks Approve. Navigates to full curation panel pre-filled with the submitted data. Admin fills missing fields (domainTags, targetAudience, timeline), sets status=open, isCurated=true, and saves.
6
PUT /api/admin/listings/:id. Listing goes live. AuditLog records the full edit with diff showing the unknown→open transition and all filled-in fields.

10. Derived Field Computation
confidenceLevel and dataSourceYear are never stored in MongoDB. They are computed at read time by the API layer and attached to the response object before it is returned to the client.

10.1 confidenceLevel Logic
const now = new Date();
const oneYearAgo = new Date(now - 365 * 24 * 60 * 60 * 1000);
const twoYearsAgo = new Date(now - 2 * 365 * 24 * 60 * 60 * 1000);

if (listing.timeline.deadline && listing.timeline.deadline > now) {
  confidenceLevel = 'confirmed';  // Active deadline set for this cycle
} else if (listing.timeline.lastDeadline && listing.timeline.lastDeadline > oneYearAgo) {
  confidenceLevel = 'estimated';  // Based on last cycle (within 1 year)
} else if (listing.timeline.lastDeadline && listing.timeline.lastDeadline > twoYearsAgo) {
  confidenceLevel = 'approximate';  // Older last deadline
} else {
  confidenceLevel = null;  // No timeline data
}

10.2 dataSourceYear Logic
dataSourceYear = listing.timeline.lastDeadline
  ? listing.timeline.lastDeadline.getFullYear()
  : null;
