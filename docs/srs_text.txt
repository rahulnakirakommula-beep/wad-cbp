1. Introduction
1.1 Purpose
This Software Requirements Specification (SRS) defines the complete functional and non-functional requirements for the 
Opportunity Intelligence Platform â€” a curated, timeline-driven opportunity discovery and tracking system for 
undergraduate students. The document is intended for use by the development team, project stakeholders, and evaluators 
at VJStartups.
1.2 Scope
The system, internally codenamed Campus Opportunity Aggregator (COA), addresses a critical gap in how undergraduate 
students discover time-sensitive opportunities such as internships, research positions, fellowships, competitions, 
hackathons, scholarships, mentorships, workshops, and incubators.
The platform is explicitly not a job board. Its core differentiator is timeline visibility â€” surfacing what is 
coming, when to prepare, and what is at risk of being missed â€” replacing accidental senior-knowledge-transfer with a 
structured, always-current intelligence system.
Scope boundaries for V1:
Single institution deployment (one college, one branch enum set).
Three user types: Student, Admin, Organisation (Source).
MERN stack: MongoDB, Express.js, React (React Router v6), Node.js.
No scraper integrations (architecture is expandable â€” targeted at V2).
No OAuth in V1 (email/password authentication only).
No public submission form â€” Organisations access a dedicated webpage and Listing Manager.

2. Overall Description
2.1 Product Perspective
The Opportunity Intelligence Platform is a standalone, web-based SaaS application deployed on free-tier cloud 
infrastructure (Vercel for frontend, Render for backend, MongoDB Atlas for database). It has no integration with 
existing college ERP or placement portals in V1. The system is self-contained and communicates with external services 
only via email (Nodemailer/SMTP).
The system consists of three logical tiers:
Presentation tier â€” React SPA served via Vercel CDN.
Application tier â€” Node.js/Express REST API deployed on Render.
Data tier â€” MongoDB Atlas (M0 free cluster) with Redis for feed caching (pre-computed Recommended section).

3.1 Authentication & Onboarding (FR-AUTH)
FR-ID
Req. Name
Description
FR-AUTH-01

The system shall return HTTP 409 if the email already exists in the database.
FR-AUTH-04

POST /api/auth/login shall validate credentials via bcrypt.compare. A successful login returns a JWT (7-day expiry) 
and updates lastLoginAt. An unverified email returns HTTP 403 with a clear message.
FR-AUTH-05
Token Storage
The frontend shall store the JWT in localStorage and include it in the Authorization header for all authenticated API 
calls.
FR-AUTH-06
Onboarding â€” Step 1
The system shall present a multi-step onboarding flow. Step 1 collects the student's academic branch (enum from the 
institution's branch list) and current study year (selector, values 1â€“6).
FR-AUTH-07
Onboarding â€” Step 2
Step 2 shall display all active DomainTags grouped by category. The student shall select up to 5 tags. Tags are loaded 
from GET /api/tags?active=true.
FR-AUTH-08
Onboarding â€” Step 3
Step 3 shall display notification preference toggles (all enabled by default). The student may adjust before 
completing.
FR-AUTH-09

PUT /api/user/onboarding shall save all onboarding fields and set onboardingComplete: true. The student is redirected 
to /app/feed.
FR-AUTH-10

FR-AUTH-11

Users with status: suspended shall be blocked from accessing /app/* routes, receiving HTTP 403.
FR-AUTH-12

On login, if role === 'admin', the frontend shall redirect to /admin. If role === 'source', redirect to /org.
3.2 Feed & Discovery (FR-FEED)
FR-ID
Req. Name
Description
FR-FEED-01
Four Feed Sections
GET /api/feed/sections shall return all four sections in a single API call: Closing Soon, Recommended for You, Don't 
Miss, and Browse All. Sections are computed via four parallel DB queries.
FR-FEED-02
Closing Soon
Closing Soon shall return up to 5 open listings whose deadline falls within 7 days from the current date, excluding 
listings the user has ignored, sorted by deadline ascending.
FR-FEED-03
Don't Miss
Don't Miss shall return up to 3 listings with priority: dont-miss and status: open, sorted by deadline ascending.
FR-FEED-04
Recommended â€” Eligibility Filter
The recommendation engine shall first filter listings to those where the student's currentYear is in 
targetAudience.years (or the array is empty â€” meaning all years eligible) AND the student's branch is in 
targetAudience.branches (or empty array). Listings already saved, applied, or ignored shall be excluded.
FR-FEED-05
Recommended â€” Scoring
Each eligible candidate shall receive a score: +2 per matching domainTag with user interests, +1 if deadline is within 
30 days, +2 if priority is high, +3 if priority is dont-miss. Results sorted by score descending; top 8 returned. If 
fewer than 3 results, the year filter is relaxed and the query reruns.
FR-FEED-06
Recommended â€” Pre-computation
The Recommended section shall be pre-computed by a background job running every 15â€“30 minutes and cached in Redis 
keyed by userId. The feed endpoint shall serve from cache on load, not re-compute on demand.
FR-FEED-07
Browse All
GET /api/feed/browse shall accept query parameters: domain[] (multi-select), type[] (multi-select), stipend, location, 
page, and limit (default 20). Results shall be filtered to status in [open, upcoming]. Sort: priority descending, 
deadline ascending.
FR-FEED-08
Filter Persistence
Browse All filter state shall be persisted in URL query parameters (React Router). Page refresh shall restore filters.
FR-FEED-09
Save from Feed
The Save button on a listing card shall POST /api/activity with { listingId, status: 'saved' }. The UI shall update 
optimistically and roll back with a toast error if the API call fails.
FR-FEED-10
Ignore from Feed
The Ignore button shall POST /api/activity with { listingId, status: 'ignored' }. The card shall disappear from feed 
immediately (optimistic). Ignored listings shall not reappear in Closing Soon or Recommended.
FR-FEED-11
Dashboard Strip
The feed page shall display a tracking summary (Saved / Applied / Missed counts) sourced from GET 
/api/activity/summary.
FR-FEED-12
Notification Badge
The top navigation shall display an unread notification count badge sourced from GET /api/notifications/unread-count.
FR-FEED-13
Parallel Load
On feed page load, the three API calls (feed/sections, activity/summary, notifications/unread-count) shall fire in 
parallel.

5. External Interface Requirements
5.1 User Interface
The application shall be a single-page application (SPA) built with React and React Router v6.
The design system follows the AccentSketch theme: a professional SaaS UI with a removable hand-drawn personality layer.
The colour palette is centred on dark navy (#1B2A4A) as the primary brand colour and amber (#E6A817) as the accent.
All student-facing pages shall be responsive and usable on mobile viewports (minimum 360px width).
Markdown content (listing descriptions, guide content) shall be rendered using a client-side Markdown renderer (e.g., 
react-markdown).
5.2 API Interface
All API endpoints shall use REST conventions with JSON request/response bodies.
Authentication is performed via Bearer JWT in the Authorization header.
API versioning is not required in V1; all routes are under /api/*.
Error responses shall use standard HTTP status codes with a JSON body: { error: 'message', field?: 'field_name' }.
The API base URL for V1 is the Render deployment URL configured via an environment variable in the React app.
5.3 Database Interface
The application communicates with MongoDB Atlas via Mongoose ODM.
The connection string is stored as an environment variable (MONGODB_URI).
The Redis cache for recommendation pre-computation is accessed via the ioredis or upstash/redis client.
5.4 Communication Interface
Outbound email is sent via Nodemailer using SMTP. The sending email address and SMTP credentials are stored as 
environment variables.
Email templates are plain HTML with inline CSS for maximum email client compatibility.
No inbound email processing is required in V1.
